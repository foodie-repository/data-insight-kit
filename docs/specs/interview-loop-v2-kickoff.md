## 배경 (v1 마감 진단, CHANGELOG 발견 4 참조)

v1 아키텍처는 batch stage + 승인 게이트라서 사용자는 완성된 결과의 결재자가 된다.
spec §4(분석가 사고 절차 안내)와 §8.3(단계별 도메인 인터뷰)이 의도한 "비전문가와
함께 탐색"하는 경험은 guided intake 한 곳에만 존재한다. 과거의 checkpoint 수정
이력(8fa391a→3255cc2→bba8329→96d8b3b→800859a→f189520)은 전부 게이트 레이어
강화였고 상호작용 레이어는 미구현 — 반복 수정에도 체감이 변하지 않은 원인.

v2의 정의: **guided intake에서 검증된 패턴(질문 JSON 생성 → 상위 UI가
AskUserQuestion으로 표시 → 답변을 draft에 누적 → 충분하면 확정)을 파이프라인
중간에 이식하는 "interview loop 런타임".** spec §12.2(Domain Expert Workflow)와
§8.3(단계별 인터뷰 lifecycle)이 계약 기반이다.

## 확정해야 할 설계 결정 3가지

### D1. 인터뷰 루프의 위치

- (a) `explore` 직후 1곳: 탐색 결과·데이터 샘플을 보여주고 "무엇이 궁금한지"
  2~3문답으로 frame을 좁힌다. 최소 변경, guided-intake 패턴 그대로 재사용.
- (b) explore 후 + analyze 후 2곳: 1차 결과를 보고 "더 파볼 방향"도 문답으로.
- (c) 매 단계: 부담 크고 v1 게이트와 중복 위험.

### D2. 탐색 문답의 형태

- (a) 선택지형: kit이 "볼 만한 방향 3개 + 미리 계산한 미니 결과"를 제시하고 고른다.
  결정적이고 안전.
- (b) 자유 질문형: 사용자가 "행정동별로 보여줘" 같은 질문을 하면 kit이 미니 쿼리를
  돌려 즉석 답변. 진짜 공동 탐색이지만 v1의 결정적 코어 철학과 긴장.
- (c) 혼합: 기본은 선택지형, 각 라운드에 자유 질문 1개 허용.

### D3. 질문 수 상한

비전문가 부담 관리. 참고: guided intake는 현재 최대 3문항. 라운드당/전체 상한을
어떻게 둘지, "충분" 판정을 무엇으로 할지.

권고안(마감 세션): D1-(a) + D2-(c)로 시작해 대표 run 검증 후 확장.
