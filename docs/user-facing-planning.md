# User-Facing Planning Contract

data-insight-kit의 계획 문서는 두 독자를 분리한다.

1. **사용자용 분석 기획안**: 사용자가 "내가 원하는 분석이 맞는지" 판단하기 위한 문서.
2. **내부 실행 계획**: 에이전트가 수집, 스냅샷, 파이프라인, QA를 실행하기 위한 문서.

Plan Mode, guided intake, human checkpoint에서 사용자가 먼저 보아야 하는 것은 항상 사용자용 분석 기획안이다. 내부 실행 계획은 뒤쪽 **기술 부록**으로 두고, 기본 표시 분량은 짧게 제한한다.

## 사용자용 분석 기획안 필수 구조

다음 항목을 쉬운 문장으로 쓴다.

- **한 줄 목적**: 이번 분석이 무엇을 보기 위한 것인지.
- **답할 질문**: 대시보드와 보고서가 답해야 할 질문 2~5개.
- **이번 데이터로 볼 수 있는 것**: 데이터가 직접 지원하는 판단.
- **이번 데이터만으로 판단하지 않을 것**: 매출, 수요, 수익성, 추천 같은 과한 결론을 막는 한계.
- **분석 방향 선택지**: 사용자가 고를 수 있는 방향 2~3개와 추천안.
- **중간 확인 시점**: 데이터 샘플과 품질 요약, 분석 방향, 대시보드 구성, 보고서 구성 중 언제 다시 물을지. 대시보드 구성 확인에는 사용할 데이터/지표, 비교 기준, 추천 차트, 대안 차트, 제외/보류 이유가 들어가야 한다.
- **실행 전 준비사항**: API 키, 원천 파일, 접근 권한처럼 사용자가 알아야 하는 준비사항이나 blocker.
- **승인 선택지**: 사용자가 바로 고를 수 있는 2~4개 선택지.
- **승인 질문**: 사용자가 바로 답할 수 있는 한 문장.

`답할 질문`에는 사용자의 업무/판단 질문만 넣는다. "단순 Top-N을 넘어서려면 어떤 차트 흐름이 적합한가요?"처럼 에이전트 품질 기준에 가까운 질문은 사용자용 질문이 아니라 내부 실행 계획이나 QA 기준으로 보낸다.
사용자용 문서에는 "이전 계획은 취소하고", "최신 계약 기준"처럼 대화 이력이나 내부 버전 관리를 드러내는 문장을 넣지 않는다. 사용자는 현재 제안이 무엇인지 판단하면 되므로, 필요하면 "이번 분석은 이렇게 진행합니다"처럼 현재형으로 쓴다.
승인 질문은 "실행을 시작하면 위 선택지를 먼저 묻습니다"처럼 시스템 동작을 설명하는 문장이 아니라, "이 범위로 먼저 데이터 확인 단계부터 시작할까요, 아니면 분석 대상·범위를 바꿀까요?"처럼 사용자가 바로 답할 수 있는 선택 질문이어야 한다.

## 내부 실행 계획 노출 규칙

다음 항목은 사용자용 기획안 앞부분에 두지 않는다. 필요하면 "기술 부록" 또는 "내부 실행 계획 요약" 섹션에만 둔다.

사용자에게 보이는 내부 실행 계획은 기본적으로 5개 안팎의 짧은 bullet로 제한한다. 상세 command, schema, endpoint, pagination, QA 명령은 사용자가 요청하거나 실행자가 필요할 때만 별도 부록으로 둔다. 초보자가 분석 방향 승인 전에 읽어야 하는 문서가 기술 설계서처럼 길어지면 실패로 본다.

- run-id, 파일 경로, command
- API endpoint, pagination, smoke test, serviceKey env
- schema, manifest, source_ref, chart_spec
- QA command, exit code, BLOCK/WARN 세부
- adapter category, metric_layer, grain 같은 내부 계약명
- 단순 ranking 반복 금지, chart diversity, QA BLOCK 같은 품질 관리 기준

## 언어 규칙

사용자용 분석 기획안에는 내부 분석 용어를 그대로 노출하지 않는다.

- `data_profile` → "데이터 확인 단계"
- `analysis_strategy checkpoint` → "분석 방향 확인 단계"
- `dashboard_storyboard` → "대시보드 구성안"
- `report_outline` → "보고서 구성안"
- `checkpoint` → "중간 확인"
- `standard + mixed + data_only` → "요약 보고서, 데이터 근거만 사용"
- `deep + mixed + data_only` → "요약 보고서와 심층 검토 보고서, 데이터 근거만 사용"
- `proxy` → "간접 지표", 또는 문맥상 "공급/경쟁 구조"
- `grain` → "분석 단위"
- `source_ref`, `manifest`, `chart_spec` → 내부 실행 계획으로 이동
- 내부 지표명이나 약어 → 필요하면 사용자가 이해할 수 있는 지표 설명으로 풀어쓴다.
- 원천 컬럼명과 행정구역 코드는 본문이 아니라 방법론/부록에 둔다.

다음 표현은 사용자용 섹션에서 쓰지 않는다.

- `data_profile`, `analysis_strategy`, `dashboard_storyboard`, `report_outline`
- 내부 스크리닝 용어 또는 과도하게 단정적인 후보 표현
- `실행을 시작하면 위 선택지를 먼저 묻고`
- `이전 계획은 취소`, `이전 계획을 취소`
- `source_api_manifest`, `checkpoint_question`, `qa/validate.py`

대신 다음처럼 쓴다.

- "데이터 확인 단계", "분석 방향 확인 단계", "대시보드 구성안 확인 단계", "보고서 구성안 확인 단계"
- "추가로 확인할 대상"
- "상대적으로 두드러지는 대상"
- "이 범위로 먼저 데이터 확인 단계부터 시작할까요, 아니면 분석 대상·범위를 바꿀까요?"
- "이번 분석은 이렇게 진행합니다"
- "기술 부록에서 수집·검증 절차를 확인합니다"

## 예시

### 좋은 사용자용 기획안

```md
# 이번 분석은 이렇게 진행합니다

## 한 줄 목적
선택한 데이터에서 분석 대상의 분포, 변화, 차이, 예외를 확인해 다음 판단에 필요한 근거를 만듭니다.

## 답할 질문
- 분석 대상은 어떤 범위와 기간을 포함하나요?
- 어떤 세그먼트나 그룹 간 차이가 큰가요?
- 어떤 지표를 기준으로 우선순위나 위험 신호를 볼까요?
- 현재 데이터만으로 판단하기 어려운 것은 무엇인가요?

나쁜 질문:

- 대시보드가 단순 상위 N개 요약을 넘어서려면 어떤 차트 흐름이 적합한가요?

이 질문은 에이전트가 지켜야 할 설계 기준이지, 사용자가 데이터로 답을 얻으려는 질문이 아니다.

## 이번 데이터로 판단하지 않을 것
데이터에 없는 수요, 비용, 성과, 원인, 미래 결과는 확정하지 않습니다. 필요한 경우 추가 데이터가 있어야 말할 수 있는 판단으로 분리합니다.

## 추천 방향
먼저 지정한 데이터 범위로 샘플과 품질 요약을 확인하고, 세그먼트 비교 / 추세 변화 / 분포와 예외 탐색 중 어떤 방향을 우선할지 고르겠습니다.

## 실행 전 준비사항
- 원천 파일, API 키, 접근 권한처럼 데이터 접근에 필요한 준비사항을 확인해야 합니다.
- 인증이나 수집 한도가 막히면 대체 데이터를 만들지 않고 원천 수집 문제로 멈춥니다.

## 승인 선택지
- 추천 방향으로 진행: 지정한 범위로 데이터 확인 단계부터 시작합니다. (추천)
- 범위나 대상을 바꾸기: 분석 대상, 기간, 지역, 고객군, 제품군 같은 범위를 먼저 조정합니다.
- 목적을 다시 정하기: 현황 진단, 위험 점검, 단순 탐색 중 다른 방향을 고릅니다.
- 외부 데이터도 함께 검토: 기본 데이터만으로 부족한 판단에 필요한 보조 데이터를 따로 확인합니다.

## 질문
이 범위로 먼저 데이터 확인 단계부터 시작할까요, 아니면 분석 대상·범위를 바꿀까요?
```

### 나쁜 사용자용 기획안

```md
run-id와 endpoint를 확정하고, smoke test와 pagination을 수행합니다. source snapshot과 chart_spec을 만들고 QA BLOCK을 확인합니다.
```

위 내용은 내부 실행 계획으로는 필요하지만, 사용자가 분석 방향을 판단하기에는 부적합하다.
